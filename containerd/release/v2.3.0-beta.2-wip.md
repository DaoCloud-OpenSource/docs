# containerd 2.3 前瞻（WIP）：从 v2.3.0-beta.2 看 5 个关键变化与 LTS 信号

> 状态：WIP（基于 pre-release）  
> 最后核对时间：2026-04-20（UTC+8）  
> 说明：本文基于 `containerd v2.3.0-beta.2` 的公开信息整理。`2.3.0` GA 发布后，文中涉及发布节奏、LTS 周期与功能默认行为的表述需要二次复核。

## 摘要

`containerd v2.3.0-beta.2` 已于 `2026-04-17T18:15:35Z` 发布。虽然这是 beta 版本，但从 release note 与 `RELEASES.md` 的信号看，`2.3` 的方向已经比较清晰：在持续强化稳定性的同时，把运行时扩展性、可观测性与镜像分发/挂载链路进一步前推。

如果只看一个结论：`2.3` 很可能成为新节奏下的关键生产版本，但当前仍应以“预发布验证”心态使用，最终以 `2.3.0` GA 与 `RELEASES.md` 当时状态为准。

## 版本背景：为什么 2.3 值得提前关注

- `v2.3.0-beta.2` 是 pre-release，不建议直接作为生产最终基线。
- 当前 release note 已明确：自 `2.3` 起，containerd minor 发布节奏与 Kubernetes 对齐，约每 4 个月一个 minor。
- 同一 release note 同时给出 LTS 信号：`2.3` 是该节奏下首个年度 LTS，计划支持至少两年。
- 在 `RELEASES.md` 当前状态中，`2.3` 仍标注为 `LTS (future)`，并带有 `tentative` 时间窗口。

这组信息组合起来的含义是：方向已明确，但“最终落板”要看 GA。

## 5 个最值得关注的变化

### 1) Shim 启动协议重构：`shim bootstrap protocol`

`2.3` 引入统一 bootstrap 协议（`BootstrapParams/BootstrapResult`），把 containerd 与 shim 的启动握手从“多通道隐式传参”收敛为“可版本化协议”。

过去常见的问题是：CLI 参数、环境变量、stdin、spec 文件等路径并存，扩展和兼容成本高。新的协议化设计更利于后续演进。

对生产环境的直接价值：

- shim/containerd 接口更稳定，隐式耦合更少。
- 未来新增能力（例如 sandbox 相关方向）扩展路径更清晰。
- 可在兼容旧路径的前提下渐进迁移。

### 2) 可观测性增强：Trace 与日志终于能“对上号”

`2.3` 在链路观测上做了两步关键补齐：

- 支持将 OpenTelemetry Trace ID 注入日志字段（可配置）。
- 插件客户端发出的出站 RPC 传播 trace 上下文。

这意味着排障链路从“日志猜测 trace”变成“日志直达 trace，再回看调用链”，尤其适合多插件、多组件协同时的故障定位。

### 3) 镜像与存储链路强化：EROFS 组合升级

围绕 EROFS，`2.3` 形成了比较完整的一组增强：

- 新增 EROFS 原生 layer media type：`application/vnd.erofs.layer.v1`
- 支持 zstd-wrapped EROFS layer：`application/vnd.erofs.layer.v1+zstd`
- 使用 `fsmount` 相关 API 规避传统 `mount()` 在复杂参数场景中的 `PAGE_SIZE` 限制问题

这组改动同时作用于分发效率、运行时挂载可用性和存储路径的可维护性，属于“系统性增强”而不是单点优化。

### 4) CRI 补齐隔离模型组合：`hostNetwork + user namespace`

`2.3` 在 CRI 中支持容器同时使用 `hostNetwork` 与 `user namespace`。该能力与 Kubernetes 相关演进方向一致。

对使用者的意义在于：可在一些对网络能力有强需求的场景下，仍保留 user namespace 带来的隔离收益，减少“只能二选一”的落地阻力。

### 5) 容器文件系统 copy 能力前置：`transfer types`

`2.3` 新增容器文件系统 copy 的 transfer 类型，为类似 `docker cp` 的体验提供标准化基础。

关键点不是“能拷贝文件”本身，而是当运行时不适合直接访问容器挂载点时，仍可通过 transfer 框架完成数据进出。这为后续插件与客户端能力扩展留下统一接口。

## 2.3 的 LTS 信号：当前可以怎么表述

截至 `2026-04-20`，公开信息已多次指向 `2.3` 的 LTS 定位，但正式对外表达仍建议保留 GA 前提：

- 事实层：release note 与发布策略文档均给出年度 LTS 和两年支持的明确信号。
- 状态层：`RELEASES.md` 当前仍是 `LTS (future)`，并标注 `tentative` 时间窗口。

建议写法：

`containerd 2.3 已被官方发布策略与预发布说明多次指向 LTS 方向，预计支持周期约两年；最终以 2.3.0 GA 发布信息与 RELEASES.md 最终状态为准。`

## 生产用户现在该做什么（GA 前）

建议在预发布环境优先完成以下验证：

1. shim 兼容性与启动链路（重点看自定义 runtime/shim 组合）。
2. Trace 与日志联动（确认 trace ID 注入与跨插件传播行为）。
3. EROFS 路径收益（镜像拉取/解包性能、兼容性、回退策略）。
4. `hostNetwork + user namespace` 组合验证（按业务安全基线验收）。
5. 文件 copy transfer 行为验证（尤其是受限挂载访问场景）。
